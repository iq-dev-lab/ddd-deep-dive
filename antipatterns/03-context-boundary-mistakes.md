# Bounded Context 경계 실수 — 너무 세분화 vs 너무 크게

---

## 🎯 핵심 질문

이 문서를 읽고 나면 다음 질문에 답할 수 있습니다.

- Context를 너무 세분화하면 분산 모놀리스가 어떻게 발생하는가?
- Context가 너무 클 때 Ubiquitous Language가 무너지는 방식은?
- Context 경계를 재조정하는 전략과 판단 기준은?
- "마이크로서비스 = 마이크로 DDD Context"라는 오해는 어떻게 바로잡는가?
- 팀 구조가 Context 경계 결정에 어떤 영향을 미치는가?

---

## 🔍 왜 이 개념이 실무에서 중요한가

DDD를 배운 후 모든 개념을 별도 Context로 만들려는 충동이 생긴다. "쿠폰도 Context, 포인트도 Context, 알림도 Context..." 결과는 하나의 화면을 그리기 위해 10개의 서비스를 호출하는 분산 모놀리스다. 반대로 하나의 거대한 "모든 것" Context는 Ubiquitous Language를 오염시킨다.

올바른 Context 경계는 이 두 극단 사이의 균형이다.

## 😱 흔한 실수 (Before)

```
과잉 분리: 너무 세분화된 Context

  CartService        → 장바구니
  CartItemService    → 장바구니 항목 (별도 서비스!)
  WishlistService    → 위시리스트
  OrderService       → 주문
  OrderItemService   → 주문 항목 (별도 서비스!)
  PaymentService     → 결제
  CouponService      → 쿠폰
  PointService       → 포인트
  AddressService     → 배송지
  NotificationService→ 알림

  "주문 목록 조회" API:
  → CartService (장바구니 수) + OrderService (주문) +
    OrderItemService (항목) + PointService (포인트) +
    ShipmentService (배송) = 5회 호출
  → 하나 다운 → 목록 페이지 실패
  → 분산 모놀리스 완성

과소 분리: 너무 큰 Context

  CommerceService 하나에 모두:
    상품, 주문, 결제, 배송, 회원, 쿠폰, 포인트, 리뷰, 알림...

  "상품"의 의미가 Context 내에서 혼재:
    상품 = 카탈로그 항목 (마케팅팀)
    상품 = 재고 단위 (창고팀)
    상품 = 주문 항목 스냅샷 (주문팀)
  
  모두 "Product"라는 같은 이름 → 팀 간 언어 혼란
  상품팀이 재고 코드를, 주문팀이 카탈로그 코드를 이해해야 함
```

## ✨ 올바른 접근 (After — 적절한 Context 경계)

```
Context 식별 기준 적용:

  쇼핑 Context:
    장바구니, 위시리스트, 장바구니 항목
    → 하나의 Context (쇼핑 경험으로 묶임)
    → 동일 팀이 관리, 강한 응집성

  주문 Context:
    주문, 주문 항목 (OrderLine은 Order 내부 Entity)
    → 하나의 Context

  결제/정산 Context:
    결제, 쿠폰, 포인트 (할인 정책 묶음)
    → 비즈니스 규칙의 응집 (할인 = 결제의 일부)

  물류 Context:
    배송, 재고, 반품
    → 물류팀이 담당, 물류 언어 공유

  회원 Context:
    회원 정보, 주소록, 등급
    → 고객 정보 응집

  커뮤니케이션 Context:
    알림(이메일/SMS/푸시) 통합
    → 채널 무관 알림 관리

결과: 10+개 → 6개로 통합
  각 Context는 하나의 팀/스쿼드가 독립 운영
  Context 간 이벤트 또는 API 통신
```

## 🔬 내부 동작 원리

### 1. 분산 모놀리스의 발생 메커니즘

```
분산 모놀리스 증상:

정의: 별도 서비스로 나뉘었지만 실제로는 강하게 결합된 시스템
     서비스 독립 배포 불가 → A 배포 시 B, C도 함께 배포해야
     하나 장애 → 전체 장애

발생 원인:

  원인 1: 화면 기반 서비스 분리
    "이 화면에 필요한 것들" → 별도 서비스
    모든 화면이 모든 서비스에 의존 → 서형 결합

  원인 2: 데이터 기반 서비스 분리
    "이 테이블 → 이 서비스" → DB 테이블 수만큼 서비스
    하나의 비즈니스 흐름이 10개 서비스 호출

  원인 3: 도메인 지식 없는 분리
    "마이크로서비스 = 작은 서비스" 오해
    비즈니스 개념 단위가 아닌 기술 단위로 분리

분산 모놀리스 감지:
  "서비스 A를 배포하려면 서비스 B도 함께?" → 분산 모놀리스
  "단순 조회에 5개 서비스 호출?" → 분산 모놀리스
  "서비스 A의 스키마가 변하면 서비스 B 코드도 수정?" → 분산 모놀리스
  "서비스 B 없이 서비스 A를 테스트할 수 없음?" → 분산 모놀리스
```

### 2. Ubiquitous Language 붕괴 패턴

```
너무 큰 Context에서 언어 붕괴:

하나의 CommerceContext에서 "상품"의 의미:
  마케팅팀: 상품 = 카탈로그 항목 (이름, 설명, 이미지)
  창고팀:   상품 = SKU (바코드, 위치, 수량)
  주문팀:   상품 = 주문 스냅샷 (주문 시 가격, 주문 시 이름)
  배송팀:   상품 = 패키지 (무게, 크기)

문제:
  Product 클래스가 모든 의미를 담으려 함
  → @Transient 필드 폭발 (어떤 상황에서 채워지는지 불명확)
  → 마케팅팀이 바코드 필드가 왜 있는지 모름
  → 창고팀이 이미지 URL 필드가 왜 있는지 모름

  "이 Product 클래스의 category 필드가 null이면 어떻게 해야 해?"
  → 아무도 모름 (어떤 컨텍스트에서 null인지 불명확)

Context 분리 후:
  CatalogItem: 마케팅팀 언어 (이름, 설명, 카테고리, 이미지)
  StockUnit:   창고팀 언어 (SKU, 위치, 수량, 배치)
  OrderLineSnapshot: 주문팀 언어 (주문 시 이름, 주문 시 가격)
  ShipmentItem: 배송팀 언어 (무게, 크기, 취급 주의)

각 팀이 자신의 언어로 작업 → Ubiquitous Language 유지
```

### 3. 경계 재조정 전략

```
Context 경계 재조정이 필요한 신호:

분리가 너무 세밀한 경우:
  □ 두 서비스가 항상 함께 배포됨
  □ 하나의 비즈니스 트랜잭션이 3개 이상 서비스를 횡단
  □ 두 서비스 간 동기 API 호출이 99%
  → 두 Context를 하나로 통합

분리가 너무 큰 경우:
  □ 같은 용어가 Context 내에서 다른 의미로 사용됨
  □ 한 팀의 변경이 다른 팀의 코드에 영향
  □ 코드베이스가 하나인데 10명 이상이 충돌
  → Context를 더 세밀하게 분리

재조정 방법:
  통합: Strangler Fig로 하나를 다른 하나로 흡수
        먼저 API를 위임, 이후 코드 합치기

  분리: 새 Context를 레거시 옆에 구현
        이벤트 구독으로 데이터 동기화
        점진적 트래픽 이전
```

### 4. 마이크로서비스 ≠ 마이크로 Context

```
흔한 오해:
  "마이크로서비스 = 작은 서비스 = 하나의 기능만"
  → 100개 서비스 → 분산 모놀리스

올바른 이해:
  Bounded Context가 배포 단위(마이크로서비스)가 될 수 있음
  하나의 Context가 여러 서비스로 구현될 수 있음
  하나의 서비스에 여러 Context가 있을 수 있음 (모놀리스 시작)

예시:
  주문 Context → 하나의 마이크로서비스 (Order Service)
  물류 Context → 하나의 마이크로서비스 (Logistics Service)
  (재고 + 배송 + 반품이 하나의 서비스)

  처음에는: 모놀리스에 모든 Context
  성장 후: Context 단위로 분리 → 마이크로서비스

Context 크기 기준:
  "두 명의 피자 팀이 독립적으로 개발하고 배포할 수 있는가?"
  → YES: Context 크기가 적절
  → NO (너무 큼): 분리 고려
  → NO (너무 세밀해서 혼자 배포 불가): 통합 고려
```

        .that().resideInAPackage("..shared..")

## 📊 설계 비교

```
너무 세분화 vs 적절한 경계 vs 너무 크게:

                너무 세분화           적절한 경계          너무 크게
────────────┼──────────────────┼──────────────────┼──────────────────
Context 수   │ 20~50개           │ 5~10개           │ 1~3개
────────────┼──────────────────┼──────────────────┼──────────────────
언어 명확성  │ Context당 명확     │ Context당 명확   │ 언어 오염
────────────┼──────────────────┼──────────────────┼──────────────────
화면 조회   │ N개 서비스 호출    │ 1~3개 호출       │ 단일 호출
────────────┼──────────────────┼──────────────────┼──────────────────
독립 배포   │ 이론상 가능, 실제 의존│ 독립 배포 가능  │ 전체 재배포
────────────┼──────────────────┼──────────────────┼──────────────────
장애 격리   │ 낮음 (연쇄 호출)  │ 높음             │ 없음
────────────┼──────────────────┼──────────────────┼──────────────────
팀 자율성   │ 낮음 (조율 많음)  │ 높음             │ 팀 간 충돌 많음
```

## ⚖️ 트레이드오프

```
경계 재조정의 비용:
  Context 통합: 코드 합치기 + 언어 통일 + DB 마이그레이션
  Context 분리: 새 서비스 운영 + 데이터 이전 + 팀 재편

현실적 조언:
  불확실할 때는 크게 시작 (모놀리스)
  도메인 지식이 쌓이면 자연스럽게 경계가 보임
  그 경계를 코드에 먼저 반영 (패키지 분리)
  이후 서비스 분리 (배포 경계)
  "먼저 Context를 찾고, 나중에 서비스로 분리"
```

## 📌 핵심 정리

```
Context 경계 실수 핵심:

너무 세분화 문제:
  분산 모놀리스 (모든 기능이 여러 서비스 호출)
  독립 배포 불가 (실질적 의존 많음)
  해결: 항상 함께 배포되는 것 → 하나의 Context

너무 큰 문제:
  Ubiquitous Language 오염
  팀 간 코드 충돌
  해결: 언어가 다른 개념 → 별도 Context

판단 기준:
  "두 개념이 항상 같이 배포되는가?" → 통합 신호
  "같은 단어가 다른 의미로 쓰이는가?" → 분리 신호
  "두 명 피자 팀이 독립 운영 가능한가?" → 적절한 크기

마이크로서비스:
  Context ≠ 서비스 (비율은 1:1부터 N:1까지 다양)
  먼저 Context → 나중에 서비스로 분리
```

## 🤔 생각해볼 문제

**Q1.** 현재 5개 마이크로서비스가 있는데, 하나의 비즈니스 요구사항이 항상 3~4개 서비스를 동시에 수정해야 한다. 이것이 분산 모놀리스라면 어떻게 해결하는가?

<details>
<summary>해설 보기</summary>

**함께 변경되는 서비스들을 하나의 Context로 통합합니다.**

```
진단:
  서비스 A, B, C가 항상 함께 변경됨
  → A, B, C는 사실 하나의 Context

해결 전략 (Strangler Fig):
  1. A, B, C를 하나의 레포지토리로 통합 (모노레포)
  2. 내부적으로 패키지로 구분 (Context 경계 유지)
  3. 단일 배포 단위로 통합
  4. 이후 다시 도메인 지식이 쌓이면 진짜 경계 재식별

단기:
  A, B, C를 하나의 서비스로 통합 (배포 단순화)

중기:
  통합된 서비스 내에서 도메인 지식 축적
  진짜 Context 경계가 보이면 다시 분리

원칙: "독립 배포 가능성"이 마이크로서비스의 핵심
     독립 배포가 안 되면 마이크로서비스의 이점이 없음
```

</details>

**Q2.** "팀이 8개이고 Context가 6개다. 각 팀이 하나의 Context를 담당해야 하는가?" 라는 질문에 어떻게 답하는가?

<details>
<summary>해설 보기</summary>

**팀과 Context가 1:1일 필요는 없습니다. Conway의 법칙을 역으로 활용합니다.**

```
Conway의 법칙: "시스템의 구조는 조직의 커뮤니케이션 구조를 따른다"

역 Conway 전략: "원하는 시스템 구조를 먼저 정하고, 조직을 그에 맞게"

예시:
  팀 8개, Context 6개:
  
  팀 1: 주문 Context (핵심)       → 3명
  팀 2: 상품 Context (핵심)       → 3명
  팀 3: 결제 Context (지원)       → 2명
  팀 4: 물류 Context (지원)       → 2명
  팀 5: 회원 Context (지원)       → 2명
  팀 6: 플랫폼 (인프라/공통)       → 3명
  팀 7: 데이터 (분석/ML)          → 2명
  팀 8: 모바일 (클라이언트)        → 3명

  팀 7, 8은 Context 소유자가 아닌 소비자
  팀 6은 Generic Context 지원

실용: Context 수와 팀 수가 일치하지 않아도 됨
     핵심 Context(Core)에 더 많은 팀/인력 투자
     Generic Context는 소수로 또는 외부 서비스로
```

</details>

**Q3.** 스타트업에서 5명 팀으로 시작하는데, 처음부터 몇 개의 Context로 나눠야 하는가?

<details>
<summary>해설 보기</summary>

**처음엔 모놀리스 안의 패키지로 시작하고, Context는 나중에 분리합니다.**

```
5명 팀의 현실적 접근:

Phase 1 (0~6개월): 모놀리스 + 내부 패키지 분리
  하나의 서비스 + 코드 내 Context 경계
  
  com.example/
    order/       ← 주문 패키지 (논리적 Context)
    catalog/     ← 상품 패키지
    member/      ← 회원 패키지
    payment/     ← 결제 패키지

  장점: 배포 단순, 도메인 지식 축적 가능
        패키지 간 경계로 의존성 관리 (ArchUnit)
        나중에 분리하기 쉬운 구조

Phase 2 (6개월~): 트래픽/팀 성장에 따라 서비스 분리
  "이 패키지의 변경 속도/요구사항이 다른 것과 다른가?"
  → YES: 별도 서비스로 분리
  
  팀이 10명이 되면: 주문 vs 상품 팀으로 분리
  → 각자 독립 서비스로

핵심: Context 경계(도메인)를 먼저 찾고, 서비스 분리(인프라)는 나중에
      경계 없이 분리하면 분산 모놀리스 → 더 나쁨
```
